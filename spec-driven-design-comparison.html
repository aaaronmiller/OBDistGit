Spec-Driven Design Systems: Ghost AI, Spec Kit, OpenSpec, Kiro, and Strata

Executive conclusion

The strongest existing system in this comparison is GitHub Spec Kit. It is the best general-purpose baseline because it defines a portable spec-driven development lifecycle, gives the lifecycle concrete command names, produces Markdown artifacts for each phase, and treats agent integration as an explicit design target rather than an accidental side effect. Spec Kit is not perfect. Its strength can become ceremony, and its default process can feel broad rather than opinionated. But as a platform for serious spec-driven design, it has the best combination of methodology, templates, governance, portability, and ecosystem surface.

OpenSpec is the strongest change-management complement. It feels more operational than Spec Kit because its proposal and archive vocabulary maps directly onto the reality of long-lived software: an idea is explored, proposed, continued, verified, and eventually archived. That makes OpenSpec especially useful when the problem is not only "write the first spec" but "keep many evolving specs coherent across time." If Spec Kit is the best base, OpenSpec supplies the missing muscle for change lifecycle and archive hygiene.

Kiro is the strongest productized user experience. Its repository evidence and public positioning show a mature spec design component baked into an agentic IDE and CLI product. The .kiro/specs pattern, requirements/design/tasks loop, and IDE-first orientation are important because most methodology fails when it is not integrated into the place where engineers actually work. Kiro's weakness, for this comparison, is that the public repository is more product/documentation surface than a fully open implementation of the spec engine. It demonstrates the workflow and validates the product shape, but it is less inspectable as reusable infrastructure.

Ghost AI is not a full SDD methodology in the same category as Spec Kit or OpenSpec. It is an interactive systems architecture builder. That matters because a better spec-driven design system should not be only files and slash commands. It should help users see the system before implementation. Ghost AI contributes the missing visual architecture surface: collaborative canvas, AI-generated specs, system graph context, and architecture-first review. Its limitation is the reverse of Spec Kit's: it is product-specific and visually strong, but weaker as a general-purpose governance and change-lifecycle methodology.

Strata is the most interesting raw material for a better version because it already encodes the user's hard-won preference: structured thinking, deliberative refinement, stage gates, and practical agent handoff. It appears less mature as a packaged CLI or IDE product, but stronger as a philosophy for forcing intent clarity and avoiding vibe-coded output. The better version should not discard Strata. It should make Strata the governance and adversarial-review layer on top of a Spec Kit style artifact spine, an OpenSpec style change lifecycle, a Kiro style product workflow, and a Ghost AI style architecture canvas.

The best answer is therefore two-tiered. If choosing an existing system today, choose Spec Kit as the primary baseline and borrow OpenSpec for changes, Kiro for UX expectations, and Ghost AI for architecture visualization. If building a better version, build a Strata-derived system that treats spec-driven design as a living loop: intent capture, requirements, context, design, tasks, implementation handoff, validation, archive, and retrospective learning. The superior system is not "more prompts." It is a governed artifact pipeline with visible state, explicit review gates, and a diagrammatic model of what is changing.

What the tables show

The main pattern is that the systems are not direct substitutes. They occupy different layers of a spec-driven design stack. Spec Kit is the artifact spine. OpenSpec is the change-lifecycle engine. Kiro is the product experience. Ghost AI is the architecture visualization surface. Strata is the deliberative governance layer. Treating them as one-dimensional competitors hides the important lesson: the best future system is compositional.

Spec Kit earns the broadest feature coverage because it does the unglamorous work required for a reusable methodology. It defines a lifecycle, supports multiple agents, exposes installation and initialization, and treats templates and integrations as first-class design objects. That is why it is the best default choice when the goal is to install something today and have a disciplined spec-to-plan-to-task workflow.

OpenSpec's high score in change lifecycle matters because most spec systems fail after the first happy path. They can create a requirements file, but they do not help maintain a living map of proposed, active, verified, and archived changes. OpenSpec's propose and archive vocabulary is closer to actual product development, where change is continuous and every alteration should preserve evidence.

Kiro's strengths are different. Kiro makes spec-driven design feel like a native mode of the development environment. Requirements, design, and tasks are not merely files to be manually assembled. They are a product flow. That matters because the hardest part of SDD adoption is not convincing people that specs are good. It is making the spec process feel cheaper than improvising.

Ghost AI brings a surface the others under-emphasize: architecture as an interactive visual object. A spec-driven design system that cannot visualize system shape leaves too much burden on prose. Ghost AI's architecture builder suggests that a better SDD system should include diagrams and canvases as first-class artifacts, not only generated appendices.

Strata's table position is the most nuanced. It does not look like the most mature package. It has fewer files, less CLI surface, and less productized onboarding. But its strength is the quality of its thinking loop. It forces a more adversarial process: define intent, structure the problem, inspect alternatives, refine, and only then execute. That is exactly the behavior missing from many spec tools when they become templates without judgment.

Feature-by-feature analysis

Intent capture

Intent capture is the first failure point. If the user starts by naming a framework, a file, or a rough implementation, a weak spec system simply converts that local wording into tasks. A strong system asks what outcome is being pursued, what failure would look like, and what constraints cannot be violated. Spec Kit and OpenSpec both provide a structured path away from raw prompting, while Strata is strongest philosophically because it treats intent as something that has to be interrogated. Kiro makes intent capture usable through product flow. Ghost AI captures intent through visual architecture rather than pure prose.

Requirements artifacts

Requirements are the first durable artifact. They should not contain framework decisions unless the requirement truly depends on them. The best requirements layer states actors, scenarios, acceptance criteria, non-goals, and failure modes. Spec Kit is the strongest general baseline here because requirements become the input to plan and tasks. Kiro's requirements/design/tasks pattern is also mature, especially for IDE users. OpenSpec gives requirements a change context. Strata should preserve its intent-first discipline but add a more standardized requirements schema.

Design artifacts

Design is where spec-driven systems often collapse into either vague architecture or premature implementation. The design artifact should translate requirements into interfaces, data models, boundaries, risks, and constraints. It should not become a dumping ground for everything the model wants to build. Ghost AI contributes heavily here because architecture visualization can expose design flaws that prose hides. Spec Kit gives the most portable design process. Kiro gives the most product-native design surface. Strata should borrow Ghost AI's diagram-first thinking and Spec Kit's artifact discipline.

Task decomposition

Tasks are where the spec becomes executable. Strong task decomposition must include dependencies, testability, and completion evidence. Weak task decomposition turns every noun in the spec into a checklist item. Spec Kit is strong because task generation is a named phase. OpenSpec is strong when tasks live inside a change lifecycle. Kiro is strong for product use because tasks sit in the same spec folder as requirements and design. Strata needs stronger durable task syntax if it is going to compete as a tool rather than a methodology.

Governance and constitution

Governance determines whether a system remains coherent across many agents. Spec Kit's constitution concept is the best named mechanism in the comparison. Strata has governance energy, but not the same packaged command and artifact surface. OpenSpec's contribution is process governance through proposal rules. Kiro appears less inspectable on this axis from the public repository, but its product likely enforces workflow structure in the IDE. Ghost AI has project instructions and agent skills, but governance is not its main thesis.

Traceability

Traceability is the core of serious SDD. A requirement should link to design decisions, tasks, implementation evidence, tests, and archive records. Without this, the spec is theater. Spec Kit and OpenSpec are strongest because they are built around artifacts and process transitions. Strata should make traceability explicit by adding IDs across intent, requirements, context, design, tasks, and evals. Kiro demonstrates the folder structure, but the public repo gives less evidence of a complete traceability engine. Ghost AI can improve traceability by linking canvas nodes to requirements and generated specs.

Validation gates

A spec system without validation gates lets the agent claim success too early. Gates should validate the spec before planning, the design before tasking, the tasks before implementation, and the implementation before archive. Spec Kit's quality checklists and OpenSpec's verify posture are strong foundations. Strata's adversarial review instinct is arguably the best qualitative pattern. The better version should combine all three: Spec Kit's artifact gates, OpenSpec's lifecycle verification, and Strata's deliberative judge.

Change lifecycle

Most real work is change work. New features are proposed, scoped, revised, implemented, verified, and archived. OpenSpec is strongest here because its vocabulary is built around changes rather than only initial project creation. Spec Kit has extension and workflow capacity, but OpenSpec feels closer to the daily lifecycle of evolving codebases. Strata should import this idea directly: every feature should have a living change record that survives implementation and becomes archive memory.

Agent portability

A better SDD system should not depend on one model, one IDE, or one CLI. Spec Kit is strongest because it explicitly targets many integrations. OpenSpec is also strong because it is oriented toward AI coding assistants generally. Strata should remain agent-agnostic but needs a distribution format. Kiro is productized around its own environment, which is a strength for UX and a limitation for portability. Ghost AI is an application, not a general integration substrate.

Extensibility

Extensibility is where a methodology becomes infrastructure. Spec Kit has the strongest extension and preset story. OpenSpec's profiles and workflow expansion provide a narrower but practical extension model. Strata can become extensible if it is packaged as templates, validators, commands, and adapters rather than prose alone. Ghost AI's skill and agent folders show a different form of extension: product-specific agent behavior. Kiro's hooks and powers suggest a mature product extensibility story, though public implementation details are limited.

Human review cadence

A spec-driven process must slow down at the right moments. Fully autonomous spec-to-code loops burn intent. The best cadence is checkpointed: approve requirements before design, approve design before tasks, approve tasks before implementation, and verify completion before archive. Strata's philosophy is strongest here. Spec Kit and OpenSpec provide the artifact checkpoints. Kiro provides product-level affordances. Ghost AI can improve by making review states visible on the canvas.

Which system is best

GitHub Spec Kit is the best existing general-purpose system. It wins because it is the most complete open toolkit for portable spec-driven development. It has the right artifact spine, agent integrations, template system, and ecosystem orientation. It is the safest baseline for a team or individual who wants an immediately usable methodology.

OpenSpec is the best system for managing ongoing change. If the comparison question is not "which tool introduces SDD best" but "which tool handles live software evolution best," OpenSpec becomes the most important competitor. Its proposal and archive ideas should be considered essential in any better version.

Kiro is the best expression of what SDD should feel like inside a product. It is less inspectable as open tooling, but the .kiro/specs pattern and IDE integration show where the UX should go. The better version should copy Kiro's product lesson: make the right workflow the path of least resistance.

Ghost AI is the best reminder that architecture is spatial. A better SDD system should use visual structure as evidence, not decoration. If a user can move from architecture graph to requirements to design to tasks, the design process becomes easier to inspect.

Strata is not the best packaged system today, but it may be the best seed for a superior one. Its advantage is not distribution. Its advantage is epistemic discipline: it cares about how the agent thinks before it writes. That is the thing the larger systems risk losing as they become command suites.

Can Strata become better than all of them?

Yes, but only if Strata stops being only a method and becomes an artifact runtime. The better version should have six layers.

  1. A Strata intent layer that forces the user to define goals, non-goals, failure modes, and decision criteria before requirements.
  2. A Spec Kit artifact layer that standardizes constitution, requirements, design, tasks, and implementation handoff.
  3. An OpenSpec change layer that tracks proposals, active changes, verification, and archive records.
  4. A Kiro UX layer that makes specs feel native in IDE and CLI workflows.
  5. A Ghost AI visual layer that turns architecture, dependencies, and user flows into reviewable diagrams and canvases.
  6. A validation layer that runs adversarial checks before completion claims and stores evidence with each artifact.

The better version should not try to be a bigger prompt. It should be a small operating system for intent. Every file should have a purpose. Every phase should have an exit criterion. Every generated task should trace backward to a requirement and forward to a verification claim. Every implementation should be able to answer: what user intent did this satisfy, what design decision enabled it, what test proved it, and what archive record explains it later?

Proposed better system: StrataSpec

The name is provisional, but the structure is clear. StrataSpec would be a spec-driven design runtime with five canonical directories:

  constitution.md
  intents/
  changes/
  specs/
  designs/
  tasks/

The intent reviewer asks whether the system is solving the right problem. The requirements reviewer asks whether acceptance criteria are testable. The design reviewer asks whether architecture is overfit. The task reviewer asks whether tasks can be independently executed. The verification reviewer asks whether completion evidence is real.

Final recommendation

Use Spec Kit now if the goal is to adopt a proven, portable SDD baseline. Add OpenSpec when changes must be managed over time. Use Kiro as the UX benchmark. Use Ghost AI as the architecture visualization benchmark. Use Strata as the philosophical core for a better system.

Build the better version by making Strata concrete: commands, templates, validators, trace IDs, artifact directories, archive records, diagrams, and gates. The best SDD system is not the one with the most files. It is the one that makes the correct next move obvious, reviewable, and hard to fake.
