Section 00

Improvement Rationale

The original v0.1 PRD was well-structured as a human-readable product document but insufficient as an AI-executable specification. In a spec-driven design workflow where coding agents implement directly from PRDs, the document must eliminate ambiguity that a human would intuitively resolve but an agent cannot.

Changes Applied

The following structural improvements were applied, each grounded in June 2026 best practices for AI-spec-driven PRDs:

Every section in this document is written so that a competent coding agent can implement it without asking clarifying questions. If a section still requires human judgment, it is explicitly flagged as REQUIRES:HUMAN.
Section 01

Executive Summary

ClipFoundry Shot Coverage Engine is a provenance-first video-planning and asset-routing system that converts a natural-language video request into a structured, rights-safe shot coverage plan. For each shot in the requested video, the system determines the optimal source strategy:

  1. Reuse an existing rights-cleared asset from the internal library
  2. Retrieve a new asset from approved external sources, verify its rights, and index it
  3. Generate a new asset via AI generation (reserved for shots that cannot be safely sourced)
  4. Route to review when rights confidence falls below threshold
  5. Omit or rewrite when no safe option exists

The system is not a stock footage library. It is a shot coverage solver: given a video request, what shots are needed, what do we already have, what can we safely retrieve, what should be generated, what needs review, and what timeline can be assembled?

The system behaves like a caching compiler for video production. Each job grows the asset graph in the direction of actual demand, making future jobs cheaper and faster. The moat is the asset graph, rights ledger, shot taxonomy, retrieval quality, risk filtering, and coverage solver — not the footage itself.

Critical Invariant
The system must never silently use assets with unknown rights in final customer outputs. This is a non-negotiable safety constraint, not a soft preference. Violation of this invariant is a critical bug.
Section 02

Problem Statement

AI video generation is expensive, inconsistent, and wasteful. A prompt like "make a 90-second video about an Indian baker making cakes" is not directly executable — it must become a structured beat sheet, shot list, asset search plan, rights plan, and edit plan.

Current generation-first systems burn compute on shots that could be cheaply retrieved or reused. They also produce random clips, low-quality continuity, overused generic imagery, and risky content involving faces, logos, stereotypes, or unlicensed footage.

The Shot Coverage Engine solves this by:

  • Decomposing creative requests into discrete shot requirements with explicit constraints
  • Mapping each shot to the cheapest safe source via a formal coverage solver
  • Retaining all newly discovered usable assets for future jobs
  • Maintaining a rights ledger that makes every asset decision auditable
Section 03

Product Thesis

The engine should become more valuable with each job. Its library grows in the direction of actual customer demand, not random internet abundance. The system pre-indexes evergreen categories but relies on just-in-time indexing for cache misses and emerging niches.

Economic Model
First job for a niche: high retrieval + generation cost. Subsequent jobs for same niche: decreasing marginal cost as the asset graph accumulates. Break-even typically within 2–4 similar jobs depending on b-roll overlap.

System Invariants

Inv-1: Rights Safety
∀ a ∈ SelectedAssets : a.pool_status = 'green' ∨ a.review_status = 'approved'
No asset with pool_status yellow, orange, or red may appear in a final export plan unless explicitly approved by a human reviewer.
Inv-2: Reference Isolation
∀ a : a.source_type = 'reference_only' ⟹ a NOTIN SelectedAssets
Reference-only records may influence search and generation but can never be selected as final output assets.
Inv-3: Customer Privacy
∀ a : a.source_type = 'customer_upload' ⟹ a.pool_status ≠ 'green' UNLESS customer_grants_global_reuse
Customer uploads are private by default and never enter the shared green pool without explicit permission.
Inv-4: Source Priority
∀ shot : green_pool_searched(shot) BEFORE external_retrieval(shot) BEFORE generation(shot)
The system must exhaust cheaper, safer sources before escalating to more expensive or riskier ones.
Inv-5: Audit Trail
∀ decision ∈ PlanDecisions : ∃ trace(decision)
Every selection, rejection, and routing decision must be logged with the reasoning, scores, and alternatives considered.

Section 07

Domain Vocabulary

7.1 Video Request

Natural-language request + optional structured settings. See VideoJob schema in §9 for exact fields.

7.2 Beat

Semantic unit in video structure. Corresponds to one idea in the script. Example: "Show mixing".

7.3 Shot Requirement

Structured description of a specific visual need. Includes description, role, camera, mood, constraints, risk flags. See §9.4 for full schema.

7.4 Asset

An image, video clip, or generated render with full metadata. Types: image, video_clip.

7.5 Coverage

Composite score [0.0, 1.0] measuring how well an asset satisfies a shot requirement. Composed of sub-scores defined in §12.

7.6 Source Strategy (Enum)

enum source_strategy: - existing_asset # From green pool - retrieved_asset # JIT retrieval, newly cleared - generated_asset # AI generation - customer_provided # Customer upload - review_required # Orange pool, needs human - reference_informed # Ref-only used to guide generation - omit # No safe source; shot removed from plan

7.7 Rights Ledger

Provenance and allowed-use record for every reusable asset. See AssetRights schema in §9.7.

7.8 Reference-Only Record

Metadata describing an online video (URL, timestamp, embedding, caption) that cannot be used as a final asset. Informative only. Enforced by Inv-2.

7.9 Pool Status (Enum)

enum pool_status: - green # Approved for automatic use - yellow # Reference-only; never selectable - orange # Requires human review before use - red # Blocked; must not be used

green yellow orange red

7.10 License Type (Enum)

enum license_type: - owned # We own it outright - generated_by_us # We generated it - customer_provided # Customer uploaded it - public_domain # No rights restrictions - cc0 # Creative Commons Zero - cc_by # CC-BY (attribution required) - paid_licensed # Paid stock license - partner_licensed # Partner agreement - reference_only # Not reusable; metadata only - blocked # Explicitly blocked - unknown # Rights status undetermined

7.11 Commercial-Safe Predicate (Formal)

Definition
is_commercial_safe(asset) :=
  asset.pool_status = 'green'
  AND asset.rights.commercial_use_allowed = true
  AND asset.rights.modification_allowed = true
  AND asset.risk_score < 0.3
  AND asset.rights.model_release_status IN ('yes', 'n/a')
  AND asset.rights.property_release_status IN ('yes', 'n/a')
  AND NOT asset.detected_faces.any_recognizable UNLESS asset.rights.model_release_status = 'yes'
  AND NOT asset.detected_logos.any_visible UNLESS explicitly_allowed

This predicate is computable — an agent can implement it as a single function. In v0.1, "commercial-safe" was used as an informal adjective. Now it is a formal boolean function over known fields.

7.12 Shot Role (Enum)

enum shot_role: - a_roll # Primary narrative; usually generated - b_roll # Supporting visuals; prime reuse target - transition # Scene transitions - insert # Detail closeups - title # Title cards - graphic # Lower thirds, overlays - establishing # Wide/establishing shots - hero # Hero/key moment shots

Best candidate final_score ≥ usable_threshold AND pool = greenSelect asset. Shot status → covered. Flag for optional review if score < strong.
Best candidate final_score ≥ partial_threshold AND pool = greenSelect asset. Shot status → partial. Add to gap list for potential JIT retrieval.
Best candidate pool = orange AND importance ≥ highAdd to ReviewItem. Shot status → needs_review.
No candidate ≥ partial_thresholdShot status → needs_retrieval if b-roll/establishing/insert. Shot status → needs_generation if a-roll/hero.
Shot role = a_roll OR heroSkip retrieval. Go directly to needs_generation. (A-roll is almost never available as stock.)
Diversity constraint violated (same source > 40% b-roll)Deprioritize repeated source. Apply 0.7× penalty to candidates from overrepresented source.
All retrieval + generation options exhaustedShot status → omitted. Log with reasoning. Include in export as "removed from plan."
Section 12

Scoring Model — Decision Table

The v0.1 "suggested formula" is replaced with a formal, configurable scoring specification. All weights have defined ranges and defaults.

Final Score Computation

formula // Configurable weights with valid ranges and defaults weights: visual_match: { default: 0.25, min: 0.10, max: 0.40 } text_match: { default: 0.15, min: 0.05, max: 0.25 } license_safety: { default: 0.25, min: 0.15, max: 0.40 } quality: { default: 0.15, min: 0.05, max: 0.25 } continuity_fit: { default: 0.10, min: 0.00, max: 0.20 } edit_fit: { default: 0.05, min: 0.00, max: 0.15 } cost: { default: 0.05, min: 0.00, max: 0.15 } // Risk is a penalty, not a weight risk_penalty_factor: { default: 0.30, min: 0.10, max: 0.50 } // Diversity is a post-hoc penalty diversity_penalty_factor: { default: 0.10, min: 0.00, max: 0.30 } final_score = (visual_match × w.visual_match + text_match × w.text_match + license_safety × w.license_safety + quality × w.quality + continuity_fit × w.continuity_fit + edit_fit × w.edit_fit + cost × w.cost) × (1.0 - risk_score × risk_penalty_factor) × (1.0 - diversity_penalty × diversity_penalty_factor) // Invariant: weights must sum to 1.0 (validated at config load) // Invariant: final_score is always in [0.0, 1.0]

Sub-Score Computation Rules

Sub-ScoreComputed ByNotes
visual_match1.0 - cosine_distance(shot_embedding, asset_visual_embedding)Shot embedding generated from description; asset from frame sample
text_matchts_rank_cd(asset.caption_tsvector, shot_description_query)PostgreSQL full-text search rank, normalized to [0,1]
license_safetyMapping: green→1.0, orange→0.3, yellow→0.0, red→0.0. Modified by rights_confidence.Orange assets get 0.3 base, multiplied by rights_confidence
qualityAsset.quality_score (from Quality Agent)Direct pass-through
continuity_fitIf shot.continuity_group is null → 1.0. Else: 1.0 - visual_distance to other selected shots in same group.Encourages visual consistency within continuity groups
edit_fitComposite: aspect_ratio_match (0.5) + duration_adequacy (0.3) + transition_suitability (0.2)Can asset be edited to fit the shot's timing and framing needs?
costMapping: owned/cc0/public_domain→1.0, cc_by→0.9, partner→0.8, paid_stock→0.6, generated→0.4, reference_only→0.0Higher = cheaper/free
risk_scoreAsset.risk_score (from Risk Agent)Used as penalty multiplier, not additive weight
diversity_penalty0.0 if asset is first from its source. 0.5 if source already used once. 1.0 if source already used ≥ 2 times.Applied post-hoc after initial ranking
Key Difference from v0.1
v0.1 added risk_score as a positive weight (0.30), which meant higher risk increased the score. v0.2 correctly treats risk as a penalty multiplier: higher risk reduces the final score. This is a critical correctness fix.

Section 15

Acceptance Criteria (BDD)

Each user story from v0.1 is formalized as testable Given/When/Then scenarios. A coding agent can convert these directly into test cases.

AC-01: Customer Creates Video Request

Scenario: Prompt with minimal input
Given a user submits POST /api/video-jobs with prompt="Make a 60s video about baking"
And no other fields are provided
When the request is processed
Then the response status is 201
And the job duration_seconds is 60
And the job aspect_ratio is "16:9"
And the job rights_strictness is "commercial_safe"
And the job status is "draft"

AC-02: Commercial-Safe Default

Scenario: System avoids risky assets by default
Given a job with rights_strictness="commercial_safe"
And the asset library contains an asset with pool_status="orange" that is a perfect visual match
And the asset library contains an asset with pool_status="green" that is a moderate visual match
When the retriever searches for a shot
Then the green asset appears in candidates
And the orange asset does NOT appear in candidates (filtered out before scoring)

AC-03: Library Searched First

Scenario: Green pool exhausted before external retrieval
Given a job has 10 shot requirements
And 7 shots have green-pool candidates with score ≥ 0.85
And 3 shots have no green-pool candidates
When the solver runs
Then 7 shots are marked "covered" with source_strategy="existing_asset"
And 3 shots are marked "needs_retrieval" or "needs_generation"
And the AuditTrail contains entries proving green-pool was queried before JIT

AC-04: Reference-Only Isolation

Scenario: Reference-only records never selected
Given the asset library contains 5 reference-only records
When any job runs retrieval
Then zero reference-only records appear in CandidateMatch[]
And zero reference-only records appear in the final export plan

AC-05: Rights Manifest Completeness

Scenario: Every selected asset has rights documentation
Given a job is in "ready" status with 8 selected assets
When GET /api/video-jobs/:id/export is called
Then the rights_manifest array has exactly 8 entries
And each entry has non-null: source, license, commercial_use, is_generated
And each entry with attribution_required=true has non-null attribution text

AC-06: Review Blocks Export

Scenario: Unresolved review items prevent ready status
Given a job has 2 pending ReviewItems
When GET /api/video-jobs/:id/export is called
Then the response status is 409
And the error code is "REVIEW_PENDING"
And the error details include the count of pending items

AC-07: A-Roll Skips Retrieval

Scenario: A-roll shots go directly to generation
Given a shot requirement with role="a_roll"
And no green-pool candidates exist
When the solver runs
Then the shot status is "needs_generation"
And the shot status is NOT "needs_retrieval"
And the AuditTrail records the decision with reason="a_roll_requires_generation"

AC-08: Customer Privacy Default

Scenario: Customer uploads are private by default
Given a customer uploads an asset via POST /api/assets/ingest with source_type="customer_upload"
When the asset is indexed
Then the asset pool_status is "orange" (not "green")
And the asset does not appear in searches for other projects

AC-09: Scoring Risk Penalty Correctness

Scenario: Higher risk reduces final score
Given two candidates with identical sub-scores except risk_score
And candidate A has risk_score=0.1
And candidate B has risk_score=0.6
When final_score is computed with default weights
Then A.final_score > B.final_score
And the difference equals (0.6 - 0.1) × 0.30 × base_score
